# 0004. Outbox pattern для side-effects

- Статус: Accepted
- Дата: 2026-04-18

## Контекст

При смене статуса заявки нужно отправлять нотификации (email/slack/webhook).
Прямой вызов внешней системы изнутри HTTP-обработчика:
- блокирует ответ пользователю;
- при недоступности внешки ломает основной путь;
- дублирует события при ретраях клиента.

## Рассмотренные варианты

- Synchronous HTTP-call наружу. Просто, но нестабильно.
- Event bus (RabbitMQ/Kafka) из роута. Сильный overhead для текущей нагрузки.
- **Transactional Outbox**. Запись события в ту же БД-транзакцию; отдельный
  воркер забирает и публикует.

## Решение

Таблица `outbox_events` (processed_at nullable). Воркер `arq` с cron-задачей
раз в секунду забирает пачку под `SKIP LOCKED`, публикует, помечает processed_at.

Сейчас "публикация" — структурный лог; легко заменить на Kafka/HTTP/email без
изменения доменного кода.

## Последствия

- (+) Атомарность доменных изменений и событий (одна транзакция).
- (+) Внешние интеграции можно добавлять без риска для основного пути.
- (+) Легко добавить retry/backoff в воркере.
- (−) Небольшая задержка доставки (до 1 секунды) — приемлемо для нотификаций.
- (−) Нужен Redis для arq (добавлен в docker-compose).
