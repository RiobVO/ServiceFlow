# 0001. API-Key как схема аутентификации (не JWT)

- Статус: Accepted
- Дата: 2026-04-18

## Контекст

Сервис используется внутри компании и внешними интеграциями (боты, скрипты).
Сессии короткоживущие JWT-токены с refresh — усложняют жизнь интеграций,
требуют поддержку clock skew и secret rotation на клиентах.

## Рассмотренные варианты

- **JWT/OAuth 2.0**. Стандарт, широкие возможности. Но: usecase преимущественно
  machine-to-machine; refresh-логика для скриптов лишняя.
- **API-Key**. Простота, легко ротировать. Требует секьюрного хранилища.
- **mTLS**. Самое безопасное, но чрезмерно для helpdesk-нагрузки.

## Решение

API-Key. Хранится как argon2id-хеш; для поиска — префикс (первые 8 символов)
индексированной колонкой. Ротация за один POST. Для клиентов — как OAuth
static token.

## Последствия

- (+) Просто для интеграций и скриптов.
- (+) Нет проблемы clock skew / refresh.
- (−) Нет встроенного TTL — требуется ротация вручную (есть endpoint).
- (−) При компрометации одного ключа затрагивается один пользователь (не система).
