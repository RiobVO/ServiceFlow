# 0005. Синхронный SQLAlchemy-стек (пока)

- Статус: Accepted (временно)
- Дата: 2026-04-18

## Контекст

FastAPI поддерживает и sync, и async обработчики. Async-стек (asyncpg + AsyncSession)
даёт выигрыш на I/O-bound нагрузке при большом числе одновременных соединений.

## Рассмотренные варианты

- Перевести всё на async сразу. Плюсы: throughput, современный стек.
  Минусы: переписывается каждый сервис, тесты, UoW, `get_current_user`,
  зависимости; возрастает риск тонких багов при миграции.
- Оставить sync, FastAPI внутри запускает sync-хендлеры в threadpool.

## Решение

Пока держим sync-стек. Текущий нагрузочный профиль (helpdesk, до сотен RPS)
его выдерживает, connection pool настроен (pre-ping, recycle).

## Последствия

- (+) Минимум риска; тестируемость прямая.
- (−) Теряем часть производительности при >1000 RPS.
- (−) При добавлении долгих внешних вызовов из роутов нужно помнить
  про `run_in_threadpool`.

### План на будущее
Когда появится реальный рост нагрузки или интеграции с медленными внешками —
мигрировать сначала репозитории (слой к БД), затем сервисы, затем обработчики.
UoW-граница упростит переход: достаточно заменить `Session` на `AsyncSession`
внутри UoW, интерфейсы сервисов останутся прежними.
